Audit Approach Options

How should we structure this redesign? Pick the one that matches your appetite.

A

Design System First

Establish a shared theme object (CSS variables + a central theme.js) with the new C-direction palette, typography scale, and spacing. Then systematically migrate all inline styles to use it — every component gets cleaner at the same time. Fixes code quality and visual consistency in one pass.

Pros

  • One source of truth for colors/type/spacing
  • Future changes are one-line edits
  • Code quality improves across the board

Cons

  • Larger upfront work before you see visual results
  • Risk of sprawling scope
B

Calendar-First Showcase ✦ recommended

Redesign CalendarTab to the new C aesthetic — it becomes the visual target for the whole app. Extract the design tokens from that work into a theme.js, then apply them to Positions. Journal and Summary follow. Code quality improvements happen naturally as we touch each file.

Pros

  • Fastest path to visible results
  • Calendar is the highest-opportunity tab
  • Design tokens emerge from real usage, not speculation
  • Each tab is a self-contained chunk of work

Cons

  • Brief inconsistency while mid-migration (old tabs vs new)
C

Positions + AccountBar Only

Narrow scope: just redesign the parts you actually look at every day — AccountBar and OpenPositionsTab. Leave Calendar and the rest for later. Quick, low-risk, immediately useful.

Pros

  • Smallest scope, fastest to ship
  • Highest daily-use ROI

Cons

  • Leaves Calendar (biggest opportunity) untouched
  • No design system foundation
